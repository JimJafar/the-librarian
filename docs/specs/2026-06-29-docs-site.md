# Spec: In-repo docs-as-code site

**Status:** Draft for review — not yet started.
**Date:** 2026-06-29
**Origin:** Brainstorm (decide → explore-change). Marketing site + README can no
longer communicate the product's surface area; the binding constraint is
**minimising drift between documentation and functionality**.

## Objective

Stand up a documentation site that lives **in this monorepo**, ships in the
**same PR** as the code it describes, and makes its most drift-prone content
(MCP verbs, CLI, primer, slash commands, capture matrix) **generated from the
canonical sources** so divergence is a CI failure rather than a reader's
discovery. It is the single home for detailed, screenshot-backed operating
guidance across every surface the marketing page and README can't cover —
installation, the five harness integrations, the dashboard, and the CLI.

**Voice & audience.** The primary reader is a *capable but non-developer*
operator — someone running The Librarian for their team or themselves, not
reading source. The site is **prose-first**: plain, easy-to-digest explanation
of what each thing is, when you'd reach for it, and how to do it, backed by
screenshots of the real UI. The human guidance — install, per-harness setup, the
dashboard tour, the operating guides — is the **centrepiece and the
highest-value part**. The generated technical reference (the 7 verbs, schemas,
CLI flags) is a **supporting appendix**: most of those APIs are designed for
*agents* to consume, so it serves agent authors and curious developers, not the
everyday reader. This is not prospective-buyer marketing — that stays on the
external site.

## Why docs-as-code (not a wiki / in-dashboard help)

Drift is a function of the *coupling* between a doc and its source of truth,
not of where the doc is hosted. A wiki drifts exactly like the external
marketing site does — hand-maintained, decoupled, unchecked. In-dashboard help
can't reach the CLI or installation. Only an in-repo site can (a) ride the
existing "every PR updates docs / `check:release`" discipline and (b) be
*generated from* and *tested against* the canonical contracts the repo already
pins with drift-guard tests.

## Success criteria (testable)

Ordered by priority. The human guidance (1–2) is the point; the generated
reference and its machinery (4–5) guard the *secondary* technical appendix.
Most of 1–2 **cannot** be CI-gated — they're enforced by a named human reviewer
on a per-release checklist (K6), not pretended into an automated test.

1. **Guided task completion (the primary bar).** A reader who has never used The
   Librarian can, following **only this site**, (a) install it, (b) connect at
   least one harness, and (c) find what each dashboard area is for and how to
   perform its main task — without reading source, schemas, or asking for help.
   Validated by a walkthrough: a non-developer (or a fresh-eyes reviewer with no
   repo knowledge) completes install → connect → "accept a proposal" using the
   docs alone. Owner + cadence per K6.
2. **Prose-first pages.** Every Start-here, harness, dashboard-area, and
   operating-guide page **opens with plain-language prose** — what this is, when
   you'd use it, how to do the main thing — before any reference detail, and
   shows the actual UI via a current screenshot. Written for a capable
   non-developer: no unexplained jargon, short sentences, task framing not
   feature framing.
3. **Builds and deploys.** `pnpm --filter @librarian/docs build` produces a
   static `dist/`; a Cloudflare Pages project serves it; CI builds the site on
   every PR and fails the PR if the build fails.
4. **Reference parity (appendix).** The generated reference contains **exactly
   the 7 MCP verbs** (`recall`, `remember`, `flag_memory`, `store_handoff`,
   `list_handoffs`, `claim_handoff`, `search_references`), each with its name,
   tool-level description, and **every parameter's name, required/optional, type,
   and a human description sourced from the param sidecar** (K7); **every** CLI
   command across **both** CLIs (`@librarian/cli` and `@librarian/installer-cli`);
   the primer text **verbatim**; **all four** slash commands; and **every**
   harness row of the capture matrix. A test asserts these counts and names
   against the canonical sources.
5. **Drift is a failing test.** Changing a canonical source (e.g. editing a verb
   description in `packages/mcp-server/src/mcp/tools/recall.ts`, the
   `DEFAULT_PRIMER` in `packages/core/src/primer.ts`, or a param sidecar entry)
   without regenerating the reference makes `pnpm check:docs` fail, with a message
   naming the stale page and the fix command (`pnpm docs:gen`). The check runs
   **after** a build so it can never pass against stale output (K8).
6. **No broken internal links.** An **internal-only** link-check (e.g.
   `starlight-links-validator` or `linkinator` scoped to the built site) passes
   in CI; every public surface (install, each of the 5 harnesses, each dashboard
   route group, each CLI command, each operating guide) is reachable from the
   nav. External links are not network-checked in CI (they'd flake).
7. **Screenshots regenerate, never rot.** A screenshot job drives the **live
   dashboard** and emits one current image per documented route; a missing/failed
   capture fails the job. Images are captured deterministically (fixed viewport,
   web-fonts loaded, animations disabled) with a pixel tolerance, and stored so
   they don't bloat the repo or flap on antialiasing (storage decision in K4).
8. **On-brand.** The site renders in Fraunces / Newsreader / IBM Plex Mono and
   the Reading Room palette via Starlight's `--sl-*` custom properties (both the
   light *and* dark palettes from DESIGN.md), so it reads as the same product as
   the dashboard.
9. **Privacy & fail-soft preserved.** No bearer tokens, machine IDs, pairing QR
   codes, or other secrets appear in any generated page, example, or screenshot.
   Screenshots are captured against seeded **sample** data (never a real vault),
   **and** secret-bearing regions on Settings → tokens/connect are explicitly
   masked before capture.
10. **Accessibility.** The docs site itself meets **WCAG 2.1 AA** — the product's
    own bar (PRODUCT.md/DESIGN.md). An automated axe/contrast check runs in CI,
    including a contrast pass over the Reading Room `--sl-*` overrides (the
    verdigris-on-paper combination is borderline and must be verified, not
    assumed).

## Scope

**In:**
- New `apps/docs` Astro + Starlight workspace, deployed to Cloudflare Pages,
  with prettier/eslint accommodation for `.astro`/`.mdx` (plugin or an
  `apps/docs` carve-out; `.astro` is otherwise unlinted/unformatted — acknowledged).
- The information architecture below. The docs site becomes the **canonical
  home** for operational prose (K6): existing root prose (README, DEPLOYMENT) is
  **migrated in and the source thinned to a short intro + link**, not duplicated.
  The 5 integration READMEs stay full (they're consumed in plugin marketplaces) —
  the noted exception. `docs/slash-commands.md`,
  `docs/harness-capture-capability.md`, and ADRs feed the generated reference.
- A generation step + drift-guard test for the reference layer, including a
  per-parameter description **sidecar** (K7) and a source-of-truth/build-ordering
  rule (K8).
- A full Playwright screenshot pipeline (seeding harness + runner + masking +
  storage strategy) covering every documented route (K4).
- An internal link-checker and an accessibility/contrast gate in CI.
- Thin per-page links from the dashboard that deep-link **into** this site, with
  a URL/anchor-stability rule + cross-surface link guard (Phase 4).
- Thinning the marketing page's *functional* sections to links into this site
  (Phase 4) — an **out-of-tree, non-versioned** copy edit in the `codeministry`
  repo (cannot ride the monorepo's PR/version-bump model).

**Out (explicitly):**
- Migrating/rebuilding the marketing site into the monorepo. It stays plain
  static HTML in `JimJafar/codeministry` on Cloudflare Pages. (Brainstorm D5, revised.)
- Versioned/multi-version docs (single "latest" only for now).
- Internationalisation.
- Auto-generating *narrative* guides — those stay hand-written (K6).
- Adding per-parameter descriptions to the **transmitted** MCP tool schemas —
  descriptions live in the docs sidecar only, to keep agent context lean (K7).

## Generated-reference contract (natural language)

This section is the **technical appendix** — secondary to the prose guides
above, and aimed at agent authors and curious developers. Each generated page is
produced from a single canonical source; the drift-guard test regenerates and
diffs against the committed page.

- **MCP verbs** ← the `ToolDefinition` (name, tool-level description, input
  schema) of each file in `packages/mcp-server/src/mcp/tools/`, **plus** a
  co-located **param-description sidecar** (K7) supplying the human "meaning" of
  each parameter. The transmitted `inputSchema` stays lean (name/type/required
  only) so agent context isn't bloated; the rich per-parameter prose lives in the
  sidecar and appears **only** in the docs. The drift-guard asserts the sidecar
  and the schema cover exactly the same parameters (no orphans either way).
- **CLI** ← help across **both** CLIs: `@librarian/cli` (`runtime.ts` `usage()`/
  `handoffsUsage()`, `commands/auth.ts`, and the per-command flag blocks for
  backup/restore/export/migrate-data-dir) **and** `@librarian/installer-cli`
  (`server/index.ts` `serverUsage`, install/uninstall/update/doctor/autoupdate).
  One command per entry, with flags.
- **Primer** ← `DEFAULT_PRIMER` in `packages/core/src/primer.ts`, rendered
  verbatim. Note this is the **shipped default**; a deployment's live primer is
  operator-editable at `vault/primer.md`, so the page documents the default and
  says so.
- **Slash commands** ← `docs/slash-commands.md` (already canonical), included
  rather than retyped.
- **Capture matrix** ← `docs/harness-capture-capability.md`, included.

**Generator source-of-truth & ordering (K8).** The generator reads **TypeScript
source** (not built `dist/`), or `check:docs` is wired to run **after**
`pnpm build`; either way editing a source file without regenerating must fail the
check — never pass against stale output. CI runs `check:docs` after the build step.

**Decided:** generated pages are **committed** and verified by a
"regenerate-and-diff" check — it matches the existing drift-guard pattern and
keeps doc changes visible in PR diffs. `pnpm docs:gen` regenerates;
`pnpm check:docs` fails CI on drift.

## Key decisions

- **K1 — Framework: Astro + Starlight.** Official, actively maintained, built
  for docs; built-in search (Pagefind), accessible, themeable. Reading Room
  fonts via Fontsource entries in `customCss`; palette/typography via `--sl-*`
  custom properties. *Source: `withastro/starlight` customization &
  css-and-tailwind guides, retrieved 2026-06-29.*
- **K2 — Hosting: Cloudflare Pages, own project, own subdomain.** Astro defaults
  to `output: 'static'` → `dist/`; Cloudflare Pages serves it via a git-connected
  build. **Note:** unlike the marketing site (plain static HTML, separate repo),
  this builds `apps/docs` out of a **pnpm monorepo subdirectory** — a new,
  fiddlier config (root install + filtered build + output dir `apps/docs/dist` +
  `NODE_VERSION`), and the deploy is **not** exercised by repo CI. The in-repo
  build itself is auto-accommodated (`pnpm-workspace.yaml` globs `apps/*`).
  *Source: `withastro/docs` deploy/cloudflare + configuration-reference,
  retrieved 2026-06-29.*
- **K3 — Drift-guard mirrors existing pattern.** The reference parity/diff test
  sits alongside the existing cross-harness drift-guard tests and the
  tool-registry test that already pins the 7 verbs.
- **K4 — Screenshots: full automation of every documented route.** A dedicated
  **seeding harness** populates every documented route with representative,
  non-secret sample data (modelled on `apps/dashboard/e2e/fixtures.ts` +
  `global-setup.ts` — the CLI `seed` verb alone is insufficient: it inserts 2
  memories and no-ops if any exist). A Playwright **runner** drives each route,
  **masks** secret-bearing regions (tokens/machine-id/QR), and emits one
  deterministic image per route. Runs as an **e2e-class CI job** (expect a heavy,
  ~e2e-length job — accepted). Storage strategy (committed via `.gitattributes`/
  LFS vs build artifact) + pixel tolerance chosen in T3.3 to stop antialiasing
  flap. *(Chosen over a pragmatic subset — full coverage now, cost accepted.)*
- **K5 — Marketing stays external (brainstorm D5).** Functional truth lives
  here; marketing curates highlights and links in.
- **K6 — Prose: docs site canonical, repo docs thinned.** Operational prose lives
  **once**, in the docs site. README/DEPLOYMENT are thinned to a short intro that
  links in; the 5 integration READMEs stay full (marketplace-consumed) as the
  exception. Narrative prose is **hand-maintained, not auto-guarded** — so
  criteria 1–2 are enforced by a **named reviewer on a per-release checklist**
  (the human equivalent of the drift-guard), not a pretended CI test.
- **K7 — Param descriptions in a sidecar, lean transmitted schema.** Author the
  missing per-parameter descriptions, but in a co-located sidecar the generator
  reads — **not** in the `inputSchema` sent to agents, so agent context stays
  lean. Trade-off: the rich per-param prose is docs-only; agents still rely on
  the tool-level description + param names at call time. Parity-safe (the Pi
  `schema-parity` test pins names/types/required, not descriptions); the
  sidecar↔schema drift-guard keeps them in sync.
- **K8 — Generator reads source / check runs after build.** See contract — closes
  the stale-`dist` footgun that would let drift pass the check falsely.

## Phased task breakdown

Each task is independently shippable and leaves the system working. Per house
rules, **the PR that ships each phase bumps the root version and adds its dated
CHANGELOG entry** (~4 bumps; the `check:release` guard enforces it).

### Phase 1 — Shell + canonical prose + deploy

- [ ] **T1.1 — Scaffold `apps/docs`.** Astro + Starlight workspace builds in the
      monorepo; `pnpm --filter @librarian/docs build` emits `dist/`. Includes
      prettier/eslint accommodation for `.astro`/`.mdx`.
      *Accept:* build, lint, and format-check pass in CI; placeholder home renders.
- [ ] **T1.2 — Reading Room theme + a11y gate.** Fonts + `--sl-*` palette (light
      and dark) applied; axe/contrast check wired into CI.
      *Accept:* site uses Fraunces/Newsreader/Plex Mono and the Reading Room
      palette; axe passes; contrast over the overrides meets WCAG 2.1 AA (#10).
- [ ] **T1.3 — Migrate canonical prose into the IA; thin the source docs.**
      Install, first-run, the 5 harness pages, dashboard tour (placeholder
      screenshots for now), operating guides, deploy section. Root README/
      DEPLOYMENT thinned to intro+link (K6); integration READMEs left full.
      *Accept:* every nav entry resolves; no operational prose is duplicated
      between repo docs and the site.
- [ ] **T1.4 — Internal link-checker in CI.** Internal-only checker over the
      built site.
      *Accept:* a broken internal link fails CI; external links not network-checked (#6).
- [ ] **T1.5 — Cloudflare Pages project.** Git-connected monorepo build serving a
      **preview URL**; public subdomain pointed at it at the end of Phase 1
      (hostname per OQ1).
      *Accept:* a pushed change appears on the preview URL.
- [ ] **T1.6 — Finalise the docs-update DoD wording.** Once the site is publicly
      live, drop the "once it ships" conditional from the docs-update rule in
      `AGENTS.md` and `CONTRIBUTING.md` and point it at the live docs site as the
      canonical home for user-facing docs.
      *Accept:* neither file still says "once it ships"; both name the live site.

### Phase 2 — Generated reference + drift-guard

- [ ] **T2.1 — Param-description sidecar.** Author per-parameter descriptions for
      all 7 tools into a co-located sidecar (K7); add the sidecar↔schema drift
      assertion.
      *Accept:* every `inputSchema` param has exactly one sidecar description and
      vice-versa; transmitted schema unchanged; Pi `schema-parity` test still passes.
- [ ] **T2.2 — Generator + `pnpm docs:gen`.** Produces the verbs (schema +
      sidecar), both-CLI, primer, slash-command, and capture-matrix pages from
      source (K8).
      *Accept:* generated pages match reference parity (#4) — exactly 7 verbs with
      param meanings, both CLIs, primer verbatim, 4 slash commands, full matrix.
- [ ] **T2.3 — `pnpm check:docs` drift-guard, after build.** Regenerate-and-diff
      test wired into CI **after** the build step, with a teaching failure message.
      *Accept:* editing a verb description or sidecar entry without regenerating
      fails the test naming the stale page + fix command; cannot pass against
      stale `dist` (#5).

### Phase 3 — Screenshot pipeline (full automation, every route)

- [ ] **T3.1 — Seeding harness.** Populate every documented route with
      representative non-secret sample data (model on `e2e/fixtures.ts` +
      `global-setup.ts`).
      *Accept:* each documented route renders representative content, not an empty
      state.
- [ ] **T3.2 — Screenshot runner + secret masking.** Playwright drives every
      route and emits one image each; token/machine-id/QR regions masked.
      *Accept:* an image per route; secrets masked (#9); a missing/failed capture
      fails the job.
- [ ] **T3.3 — Determinism & storage.** Fixed viewport, fonts loaded, animations
      off, pixel tolerance; storage decision (`.gitattributes`/LFS vs artifact).
      *Accept:* re-running with no UI change produces no image diff (#7).
- [ ] **T3.4 — Wire into CI + embed in tour pages** (replaces Phase 1
      placeholders).
      *Accept:* runs as an e2e-class CI job; each tour page shows its current
      screenshot (#2).

### Phase 4 — Thin links (dashboard + marketing)

- [ ] **T4.1 — Stable docs URLs + cross-surface link guard.** Fix the docs URL/
      anchor scheme and add a check that deep-link targets exist before shipping.
      *Accept:* a renamed/removed docs target fails the guard rather than 404-ing
      in the wild.
- [ ] **T4.2 — Dashboard deep-links.** Per OQ2 affordance, link from dashboard
      pages into the matching docs page.
      *Accept:* links resolve to the right docs section; no prose duplicated.
- [ ] **T4.3 — Thin marketing functional sections** (out-of-tree, non-versioned,
      in `codeministry`).
      *Accept:* functional detail on the marketing page is replaced by links into
      the docs site. *(Feasibility unverified — separate repo.)*

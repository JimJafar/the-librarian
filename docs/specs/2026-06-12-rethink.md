# Spec: The Librarian Rethink (v1.0 consolidation)

**Source of truth for *why*:** `proposals/2026-06-12-rethink-brainstorm.md` (decisions D1–D16, audit, 14 scenario walks). This spec defines *what* and *how* for a single implementation run. Where this spec and the brainstorm doc conflict, this spec wins and must be amended consciously.

**Status:** Approved by owner 2026-06-12 (open questions resolved, §15). Ready for implementation — single branch, phase commits, green gates.

---

## 1. Objective

Carve The Librarian down to its converged shape:

> A markdown+git vault of three note types (memories, handoffs, references), served to any harness over MCP as 7 verbs, kept healthy by one curator with one apply rule, taught to agents by one ≤2KB primer riding the thinnest native channel per harness — with a dashboard that is the complete admin surface (vault editing, history, diffs, rollback) so operators never need git or Obsidian.

Success looks like: a fresh operator can run the server, add one MCP config block to any of five harnesses, and have working shared memory — and the codebase a contributor must understand has shrunk by roughly a third while agent-facing capability grew.

This is a **deletion-heavy refactor of a working system**, not a rewrite (D3). The vault/corpus layer, hybrid index, handoff store, and intake spine are keep-quality and must survive behaviorally unchanged except where this spec says otherwise.

## 2. Tech stack (unchanged)

- TypeScript, Node ≥22.5, pnpm 9.15 monorepo (`packages/core`, `packages/mcp-server`, `packages/cli`, `packages/intake-eval`, `apps/dashboard`)
- Vitest; Next.js 15 + React 19 + tRPC (dashboard); node-llama-cpp embeddings
- New: one Python package (Hermes adapter) under `integrations/hermes/` with its own pytest run, wired into CI

## 5. The agent contract (the spine of the system)

### 5.1 Exactly 7 MCP tools — no internal tools

| Tool | Behavior |
|---|---|
| `recall` | Hybrid search over **memories only** (keyword+vector RRF + 1-hop backlinks). No namespace filtering (D8). |
| `remember` | Fire-and-forget into intake inbox. |
| `flag_memory` | Reason required; soft-demote + review queue. Unchanged. |
| `store_handoff` | 5-section template enforced. Unchanged. |
| `list_handoffs` | Unchanged. |
| `claim_handoff` | Atomic claim, 409 on race. Unchanged. |
| `search_references` | Separate verb by design (D12); chunked + cached (§9). |

`conv_state_get/upsert/clear` are **deleted**, not hidden (D10). Update `scripts/healthcheck.js` EXPECTED_TOOLS and `packages/mcp-server/tests/mcp/tool-registry.test.ts` to pin exactly these 7 and list the newly retired verbs.

**Tool descriptions are a first-class deliverable**, not boilerplate: they are the only teaching surface guaranteed to render in every harness. Each must carry its protocol (e.g. `store_handoff`'s description contains the 5-section template; `recall`'s says "call before answering questions that may have prior context"; `remember`'s says "call when you learn a durable fact, preference, or decision"). Budget ≤1KB each.

### 5.2 The primer (D9–D11)

- Lives at `vault/primer.md`, operator-editable via dashboard vault editor; server enforces **≤2KB** on save (same rule as curator addendums).
- Served three ways from one source: (a) MCP `initialize` result `instructions` field; (b) unauthenticated `GET /primer.md` (for OpenCode's remote-URL instructions config — the endpoint must never interpolate operator-specific or secret content); (c) read by the Hermes/Pi adapters.
- Default primer content ships in the repo and must include, in priority order (truncation-safe): the recall/remember behavioral loop; the handoff protocol ("to hand off, call store_handoff; to take over, list_handoffs → claim_handoff"); the learn protocol ("when asked to extract lessons, call remember once per durable lesson"); private mode (D11: marker blocks `remember`/`store_handoff`/`flag_memory`; reads stay allowed and reach server logs); "never block the user's work if the server is unreachable."
- Wording constraint: must not pattern-match prompt-injection heuristics (no "ignore/disregard …instructions" phrasing) — Hermes regex-screens MCP-adjacent content.

### 5.3 Curator (D4, D6, D13)

One curator engine, one prompt core (versioned, replaces intake-v4/grooming-v2 pair), one operation vocabulary: `create | update | merge | split | archive | noop`.

- **Invocation modes:** on-submission (one inbox item + navigate-gathered evidence), on-schedule (corpus slice — slices are now project-key-based only, since agent-private namespaces are gone), run-now (admin).
- **One apply rule (D13):** `create/update/merge/noop` auto-apply when `confidence ≥ threshold` (single settings knob, default 0.8); `archive/split` **always** propose; any operation targeting a `requires_approval` memory proposes. Delete: `risk_level`, auto-apply policy levels, under-evaluation lifecycle, dry-run modes, the second force-propose implementation.
- **Addendums (D4):** still per-job-mode optional sections, still `.curator/*.md` ≤2KB git-committed; edits apply immediately; rollback = dashboard restore (§8). Delete `addendum_status`/`eval_version` settings and machinery.
- Curator chat survives unchanged in capability (discuss memory, propose actions, draft addendum edits) minus evaluation-state actions.
- Keep: redaction, input-hash idempotency, slice locking, bounded `max_memories`, decision log, post-intake grooming trigger with debounce.

## 6. Deletions (D2, D7, D8, D10)

Each deletion = code + tests + tRPC routes + dashboard UI + docs references. Grep for stragglers; `pnpm typecheck` is the net.

1. **Skills:** `skill-store.ts`, `list_skills`/`get_skill` tools, `vault/skills/` handling, dashboard mentions, plugin parity code.
2. **conv_state:** 3 tools, `conversation-state-store.ts` sidecar, schemas, the per-turn injection scripts in all plugins.
3. **Namespaced index:** `namespaced-index.ts` wrapper + backlink scoping; recall runs on the plain hybrid index.
4. **Classifier:** worker stub, `pendingClassification` routing, provider config, local-GGUF lifecycle, dashboard cockpit. `is_global`/`requires_approval` remain plain frontmatter booleans settable by admin/curator only.
5. **SQLite backend:** the SQLite store, dual-path `createLibrarianStore` factory, env flag, remaining SQLite-shaped tRPC contracts. Markdown is the only backend.
6. **Sessions remnants:** read-only dashboard session views, legacy bak handling, `docs/migration-sessions-storage.md` moved to an archive note.
7. **Domain field:** stop writing it; tolerate-on-read, strip-on-write; CLI `migrate-data-dir` (§10) strips it in one sweep.
8. **Event-ledger throwers:** delete `LEDGER_RETIRED` throw paths; the replacement is the git history surface (§8).
9. **Dead exports:** `VerifyResult` et al.
10. **Parked proposals:** delete `proposals/safe-fallback-capture.md` and `proposals/memory-healthchecks-and-benchmarks.md` with a CHANGELOG line; fold `proposals/hybrid-recall.md`'s relevant ideas into §9 work or delete with the same note.

**6.4 intake-eval:** rename/generalize to evaluate the unified curator prompt (judge fixtures run against the single prompt core). Keep small; if generalization exceeds ~2 focused sessions, descope to "compiles + still evaluates intake mode" and file a TODO.

## 7. Integrations (D9, D10, D14)

- Create `integrations/<harness>/` per §4 by porting the **surviving** logic from the five external repos (read them at `~/code/the-librarian-{claude,codex,opencode,hermes}-plugin`, `~/code/the-librarian-pi-extension`); external repos are archived manually by the owner afterwards (out of scope for the run).
- **Claude:** marketplace manifest + 4 command markdown files (thin restatements of primer protocols). No hooks, no scripts, no MCP client.
- **Codex / OpenCode:** README-only — exact MCP config JSON; OpenCode adds the one-line `instructions: ["<server>/primer.md"]`.
- **Hermes:** Python provider — `get_tool_schemas()` returns all 7 (verified the ABC accepts arbitrary lists), `handle_tool_call` proxies over HTTP (keep existing stdlib client + fail-soft posture), `system_prompt_block()` returns the fetched primer (cached, fail-soft empty). Delete sync_turn/on_pre_compress/on_session_end no-ops only if the ABC marks them optional (it does — they're non-abstract).
- **Pi:** extension — `before_agent_start` injects primer into systemPrompt; `pi.registerTool` × 7 proxies. Drop everything else.
- Each integration ships its own minimal tests (schema shape, render contract, fail-soft on server-down) co-located.

## 8. Dashboard (D15, D16)

**Vault explorer/editor (Obsidian-lite):**
- Tree over the entire vault (memories, handoffs, references, `.curator/`, `primer.md`).
- View: rendered markdown, frontmatter as a property table, wikilinks clickable, backlinks pane (from existing link graph).
- Edit: raw markdown editor; on save → frontmatter schema validation (reject with errors, never write invalid), then through the store layer (git commit + index invalidation). Create/rename/delete with wikilink-integrity rewrites (existing machinery). Primer/addendum saves enforce 2KB.
- Explicitly **not** v1: graph visualization, saved searches, multi-file operations.

**History / diff / rollback (this is also the audit-trail view, replacing the ledger):**
- Per-file: commit list (`git log --follow`), unified diff view, "Restore this version" → new commit reverting that file's content (never history rewrite).
- Whole-vault: activity feed of vault commits (curator commits show provenance notes); "Restore vault to this commit" → guarded flow: confirmation modal stating what changes → pause curator/intake → create pre-restore tag → revert-commit the tree state → resume. Index rebuilds from markdown by construction.
- Server-owned git (D16): on boot, if the data-dir vault is not a git repo, `git init` + initial commit automatically.

## 9. References completion (D5)

- **Persistent embedding cache:** sidecar (e.g. `<data-dir>/embeddings-cache/`) keyed by file path + content hash + embedder/model id; invalidate per file on change; survives restarts. Applies to memory embeddings too if low-cost, but references are the requirement.
- **Chunking:** split references by heading/size into overlapping chunks; index chunks; `search_references` returns the best chunk(s) with file path + heading anchor. Replaces ~2K-token truncation. A 553KB document must be searchable end-to-end.

## 10. Data-dir migration (one CLI command, never automatic deletion)

`pnpm --filter @librarian/cli migrate-data-dir` (also runs checks on server boot in warn-only mode):
- Verifies vault is a git repo (init if not — also done at boot, §8).
- Renames `consolidation-runs.json` → `intake-runs.json` (and code reads the new name with one-time fallback).
- Strips `domain` (and other retired fields) from frontmatter in one sweep commit.
- Removes retired settings keys (classifier, addendum_status, auto_apply levels, legacy curator.llm/consolidator keys).
- **Reports** (never deletes) legacy artifacts it finds: `librarian.sqlite`, `events.jsonl`, root `memories.md`, `*.predeprecation.bak`, `conv-state.json` — prints an "archivable" list with sizes.

## 11. Code style

Existing conventions govern (AGENTS.md, `docs/adr/0003-no-any.md`, eslint config, deterministic-YAML frontmatter writer, naming-canon check). Comment density and idiom must match surrounding code. New dashboard components follow the existing `apps/dashboard/components/` patterns (server actions + tRPC, no new state libraries).

## 12. Testing strategy

- Vitest co-located per package, as today. Behavior-level tests for: 7-verb registry pin; primer endpoint + 2KB enforcement + instructions field presence in initialize result; curator apply rule (each op type × confidence × requires_approval matrix); vault editor save-validation; restore flows (per-file and whole-vault, including curator-pause); embedding cache invalidation; chunked reference search; migration command idempotency.
- Hermes adapter: pytest with a fake HTTP server (port existing plugin test approach).
- The carve-down must keep `pnpm test` green at **every phase boundary** (the run's checkpoints), not just at the end.
- `check:test-count` baseline adjusted per phase as per §3.

## 13. Boundaries

**Always:**
- Keep `pnpm test`, `pnpm typecheck`, `pnpm lint` green at every phase boundary; commit per task; CHANGELOG entry per phase; conventional, focused commits.
- Route every vault write through the store layer.
- Update docs (README, AGENTS.md §tool list, slash-commands.md, DEPLOYMENT.md) in the same phase that changes the behavior they describe.

**Ask first (stop and surface to the owner):**
- Any operation touching `~/the-librarian` (the live instance) or any path outside the repo worktrees — the run operates on `~/code/the-librarian` (+ read-only on the five plugin repos).
- Publishing to npm; archiving/pushing the external plugin repos; force-push of any kind.
- Discovering that something marked for deletion has a live consumer this spec missed.
- Schema changes beyond those specified here.

**Never:**
- Delete or rewrite git history of any vault.
- Commit secrets (`check:no-secrets-in-vault` stays mandatory).
- Remove a failing test to get green — fix or consciously delete with the feature it tested.
- Touch `data/`, `data-seed/`, `data-migration/` contents anywhere.

## 14. Success criteria (the run is done when all are true)

1. `pnpm test && pnpm typecheck && pnpm lint && pnpm healthcheck && pnpm smoke` green at HEAD.
2. Tool registry test + healthcheck pin exactly the 7 verbs; `tools/list` over HTTP returns exactly 7; initialize result carries `instructions` ≤2KB sourced from `vault/primer.md`.
3. `GET /primer.md` serves the primer unauthenticated; all other admin/tRPC routes still authenticated.
4. Grep-clean: zero source references (excluding CHANGELOG/ADRs/brainstorm) to: skills store, conv_state, classifier, SQLite store, domain field, risk_level, addendum_status, under_evaluation, dry_run, namespaced index, LEDGER_RETIRED.
5. Curator: one prompt core; the D13 apply-rule test matrix passes; intake → augment and groom → merge flows work end-to-end against a fixture vault (existing e2e style).
6. Dashboard: vault tree renders the fixture vault; editing a memory through the UI produces a git commit and updated recall results; invalid frontmatter is rejected with a visible error; per-file restore and guarded whole-vault restore both work; activity feed shows curator commits with provenance.
7. References: a >100KB fixture document is searchable in its tail sections; second server start does not re-embed unchanged references (assert via embedder call count).
8. `integrations/` contains all five harness surfaces per §7, each with passing tests; pytest wired into CI for Hermes.
9. `migrate-data-dir` run twice on a legacy-shaped fixture data dir is idempotent and reports (not deletes) legacy artifacts.
10. Docs coherent: README/AGENTS.md describe the 7-verb surface and integrations; ADR 0007 records the rethink; CHANGELOG entry; version bumped to **1.0.0-rc.1**.

# Implementation plan

## Phase ordering & dependency rationale

```
P1 Carve-down + curator unification     (everything else stands on the slimmed core)
P2 Primer + 7-verb surface + integrations/   (needs P1's tool deletions)
P3 Dashboard: vault editor + history/rollback (needs P1's store-only writes & ledger removal)
P4 References: cache + chunking          (independent of P2/P3; ordered after to keep risk serial)
P5 Migration CLI + docs + ADR + release prep
```

P2 and P3 touch disjoint code (mcp-server/integrations vs dashboard) and could interleave, but the run executes serially; checkpoints between phases are full green-gate runs (§12). Biggest risks: (a) hidden consumers of deleted modules — mitigated by typecheck + grep gates and the ask-first boundary; (b) curator prompt regression from unification — mitigated by porting the intake-v4 judge fixtures into the unified prompt's eval before swapping it in; (c) whole-vault restore corrupting state — mitigated by pre-restore tag + curator pause + fixture-based e2e.
